## 1. Введение

В 2025–2026 годах профессия **AI Engineer** — одна из самых востребованных в индустрии. Но она кардинально отличается от классических ролей вроде Data Scientist или ML Engineer.

**AI Engineer — это бэкенд-разработчик будущего (и уже настоящего).**
Вы не тренируете модели. Вы **интегрируете, оркестрируете и деплоите** решения на основе готовых LLM (Large Language Models).

### 1.1 Кто такой AI Engineer? (vs ML Engineer, Data Scientist)

#### Чем занимается AI Engineer?

- Интегрирует LLM (OpenAI, Anthropic, Ollama, Groq и др.) в production-системы
- Строит RAG-приложения (Retrieval-Augmented Generation) для работы с внутренними документами
- Создаёт агентов, которые могут вызывать инструменты, принимать решения, вести диалог
- Настраивает мониторинг, логирование и безопасность LLM-вызовов
- Деплоит микросервисы на FastAPI + vLLM / Ollama / LiteLLM
- Автоматизирует бизнес-процессы через low-code (n8n, Make) + AI
- Работает с LangChain, LlamaIndex, LangGraph, CrewAI

Ваш стек: **Python, FastAPI, Docker, Qdrant, Redis, LangSmith, n8n** — а не pandas, sklearn или TensorFlow.

#### А кто такой ML Engineer?

- Тренирует и дообучает модели (fine-tuning, LoRA, quantization)
- Работает с датасетами, feature engineering, гиперпараметрами
- Использует PyTorch, TensorFlow, Hugging Face Transformers
- Занимается MLOps: эксперименты, репродюсабельность, CI/CD для моделей

Это **не ваш путь**, если вы не хотите углубляться в математику и обучение моделей.

#### А кто такой Data Scientist?

- Анализирует данные, строит дашборды, делает A/B-тесты
- Пишет SQL, работает с pandas, matplotlib, seaborn
- Часто не пишет production-код
- Фокус — на бизнес-метриках, а не на инженерии

Это тоже **не ваш путь**, если вы backend-разработчик.

### 1.2 Почему backend-разработчику это ближе, чем кажется

#### Почему именно Python-бэкендеру — проще стать AI Engineer?

Потому что:

- Вы уже умеете писать микросервисы, API, работать с базами данных
- Вы знаете Docker, Git, CI/CD, асинхронность
- Вы привыкли думать об архитектуре, масштабируемости, безопасности
- LLM для вас — просто ещё один внешний сервис, как PostgreSQL или Stripe API

Вам не нужно становиться «математиком». Вам нужно научиться **интегрировать ИИ в реальные системы**.

Если вы Python-бэкенд разработчик — вы уже обладаете **90% навыков**, необходимых для работы AI Engineer. Остальные 10% — это новые инструменты и паттерны, которые легко освоить на основе вашего опыта.

Вот почему переход для вас максимально плавный.

#### Вы уже умеете работать с внешними API

LLM (OpenAI, Anthropic, Groq, Together.ai и др.) — это просто **HTTP-API с JSON-ответами**.
Вы уже знаете:

- Как делать асинхронные запросы
- Как обрабатывать ошибки, таймауты, ретраи
- Как кэшировать ответы (Redis, in-memory)
- Как ограничивать частоту вызовов (rate limiting)

Разница лишь в том, что вместо `{"user_id": 123}` вы получаете `{"answer": "..."}`. Логика интеграции — та же.

#### Вы привыкли к production-мышлению

AI Engineer — это не эксперименты в Jupyter Notebook. Это:

- Сервисы на FastAPI / Django
- Docker-контейнеры
- Логирование, мониторинг, трассировка
- Авторизация, защита от abuse, управление секретами
- Тестирование, CI/CD, деплой

Всё это вы уже делали. Теперь просто добавляете LLM как компонент системы.

#### Вы понимаете архитектуру

RAG, агенты, оркестрация — это **паттерны проектирования**, а не магия.

- RAG = загрузка → эмбеддинги → поиск → генерация → ответ
- Агент = цикл: мыслить → выбрать инструмент → выполнить → запомнить → повторить
- Оркестрация = DAG, state machine, очереди задач

Это то же самое, что и раньше: вы строите конвейеры данных, только теперь один из шагов — вызов LLM.

#### Вам не нужно изобретать модели

Современные LLM (GPT-4o, Claude 3.5, Llama 3.1, Mixtral) уже решают 95% задач «из коробки».
Ваша задача — **грамотно их использовать**, а не улучшать.

- Нужна суммаризация? Есть готовая функция.
- Нужен парсинг JSON? Используйте `response_format={"type": "json_object"}`.
- Нужен анализ документа? Загрузите его в RAG.

Вы — инженер, а не исследователь. Ваша ценность — в **надёжной, масштабируемой интеграции**, а не в создании новой архитектуры нейросети.

#### Вы уже пишете на нужном языке

Python — основной язык AI Engineering.
Вы уже знаете:

- Pydantic для валидации данных
- Async/await для производительности
- Type hints для читаемости
- Virtualenv, pip, poetry, uv для управления зависимостями

Новые библиотеки (LiteLLM, LangChain, LlamaIndex) построены поверх этих же принципов.

#### Итог

AI Engineering — это **естественная эволюция backend-разработки**.
Вы не начинаете с нуля. Вы **расширяете свой стек** новыми инструментами, сохраняя всю инженерную дисциплину, которую уже освоили.

### 1.3 Инструменты, которые заменяют «модели»: LLM как сервис

Раньше, чтобы использовать ИИ, нужно было:

- Скачать датасет
- Выбрать архитектуру модели
- Настроить обучение на GPU
- Оптимизировать инференс

Сегодня всё изменилось. **LLM — это сервис**, как Stripe, Twilio или SendGrid. Вы вызываете API — и получаете результат. Никакого обучения, никакой математики.

Это принципиальный сдвиг. И он делает AI Engineering доступным для бэкенд-разработчиков.

#### LLM как внешний сервис

Современные LLM предоставляются через RESTful API:

- OpenAI (GPT-4o, GPT-4o mini)
- Anthropic (Claude 3.5 Sonnet, Haiku)
- Groq (Llama 3.1, Mixtral — ultra-fast inference)
- Together.ai (Llama, Mistral, Qwen)
- Fireworks.ai (оптимизированные open-source модели)

Вы отправляете JSON с промптом — получаете JSON с ответом. Всё, что нужно — `httpx` или `requests`.

#### Локальные LLM — когда нужен полный контроль

Иногда данные нельзя отправлять во внешние сервисы. Тогда используют локальные модели:

- **Ollama** — простейший способ запустить Llama, Mistral, Gemma локально
- **LM Studio** — GUI-интерфейс для локальных моделей
- **vLLM** — production-ready сервер для high-throughput inference
- **llama.cpp** — запуск моделей на CPU (даже на Macbook без GPU)

Даже в этом случае вы работаете через HTTP-API. Например, Ollama предоставляет `/api/generate` — и вы вызываете его так же, как OpenAI.

#### Единый интерфейс для всех провайдеров

Чтобы не переписывать код под каждый API, используют прослойки:

- **LiteLLM** — один вызов для 100+ моделей (OpenAI, Anthropic, Ollama, Groq и др.)
- **OpenAI-Compatible API** — многие провайдеры (Groq, Fireworks, Together) поддерживают тот же формат, что и OpenAI

Сменили провайдера — поменяли одну строку. Всё остальное — без изменений.

#### Что это даёт вам как разработчику?

- **Нет зависимости от ML-команды** — вы сами интегрируете ИИ
- **Быстрый прототип → продакшен** — один и тот же код работает в обоих режимах
- **Гибкость выбора** — сегодня Groq, завтра Claude, послезавтра локальная модель
- **Фокус на бизнес-логике**, а не на инфраструктуре модели

#### Важно: вы не выбираете "лучшую модель"

Вы выбираете **модель под задачу**:

- Нужна скорость и низкая стоимость? → GPT-4o mini, Claude Haiku, Mixtral
- Нужна глубокая аналитика? → GPT-4o, Claude Sonnet
- Нужна работа с длинным контекстом? → Claude (200K), Llama 3.1 (128K)
- Нужна полная приватность? → Llama 3.1 через Ollama или vLLM

## 2. Базовый стек AI-инженера



### 2.1 Основы LLM: что нужно знать

Прежде чем вызывать API, важно понимать, с чем вы работаете. Вам не нужно знать математику трансформеров — но нужно чётко представлять возможности, ограничения и ключевые термины, чтобы принимать инженерные решения.

#### Что нужно изучить на этом этапе

- **Что такое LLM**: большая языковая модель как статистический генератор текста, а не «база знаний»
- **Токены**: единица обработки текста (не слово!), влияние на стоимость и контекстное окно
- **Контекстное окно**: сколько «помнит» модель, как работает скользящее окно, что происходит при переполнении
- **Температура и параметры генерации**: как `temperature`, `top_p`, `max_tokens` влияют на предсказуемость ответа
- **Трансформеры (на пальцах)**: архитектура как «конвейер внимания», без формул и матриц
- **Ключевые термины**: эмбеддинг, инференс, fine-tuning, prompt, completion, system/user/assistant роли
- **Ограничения LLM**: галлюцинации, отсутствие «понимания», зависимость от формулировки промпта, задержка инференса

#### Ресурсы для изучения

- Почему контекстное окно критично для AI ассистентов и как управлять им в Cursor, Claude Code и других инструментах
- Разница между prefill и decode фазами, оптимизация затрат через кэширование и правильное использование API
- LLM vs Reasoning models vs AI агенты — три уровня сложности и когда какой использовать
- Context engineering: почему контекст важнее промтов и как структурировать информацию для агентов
- RAG, in-context learning и fine-tuning — три способа дать AI недостающие знания и когда применять каждый

#### 4. [Ключевые понятия LLM](https://habr.com/ru/companies/bothub/articles/928752/)

Краткий справочник по терминам, с которыми вы столкнётесь в документации:

## 4. RAG (Retrieval-Augmented Generation)



### 4.1 Архитектура RAG

RAG (Retrieval-Augmented Generation) — это паттерн, в котором LLM получает контекст из внешней базы знаний перед генерацией ответа. Ваша задача — понять поток данных и точки принятия инженерных решений.

#### Что нужно изучить на этом этапе

- **Базовый пайплайн RAG**: загрузка → чанкинг → эмбеддинг → индексация → поиск → генерация
- **Чанкинг**: стратегии разбиения документа (по размеру, по структуре, семантический), перекрытие чанков
- **Эмбеддинги**: что это такое, как измеряется сходство (косинус, евклид), зачем они нужны в RAG
- **Векторный поиск**: базовая идея ANN (approximate nearest neighbor), компромисс скорость/точность
- **Реранкинг**: двухэтапный поиск (векторный + cross-encoder) для повышения релевантности
- **Контекстная сборка**: как отобрать и скомпоновать чанки в промпт без переполнения окна
- **Ограничения базового RAG**: потеря контекста при чанкинге, шумные результаты, задержки

> 💡 RAG — не магия, а конвейер. Каждый этап можно измерить, оптимизировать и протестировать.

### 4.2 Векторные базы

Векторные базы данных — это хранилища, оптимизированные для работы с эмбеддингами: они позволяют быстро находить «похожие по смыслу» объекты в многомерном пространстве. Ваша задача — понять, как такие базы устроены, чем они отличаются от классических БД и какую роль играют в RAG‑системах.

#### Что нужно изучить на этом этапе

- **Эмбеддинги**: что такое векторное представление текста/объекта и зачем оно нужно для поиска по смыслу
- **Векторный поиск**: идея поиска ближайших соседей (nearest neighbors), отличие точного и приближённого поиска
- **Алгоритмы индексации**: HNSW, IVF и другие структуры, ускоряющие поиск по миллионам векторов
- **Архитектура векторной БД**: коллекции, векторы, payload (метаданные), фильтры, индексы
- **Интеграция с RAG**: как векторные базы становятся «внешней памятью» LLM и хранят знания для retrieval

> 💡 Векторная база — это «поисковый движок по смыслам», а не по ключевым словам. От её качества напрямую зависит, что модель увидит в контексте перед генерацией ответа.

### 4.3 Оценка качества RAG

RAG‑система может выглядеть убедительно, но давать неточные или нерелевантные ответы.
Ваша задача — понять, как измерять качество RAG по отдельным шагам (retrieval и генерация) и по системе в целом, чтобы уметь её улучшать, а не «тюнить вслепую».

#### Что нужно изучить на этом этапе

- **Оценка качества ответа**: релевантность, достоверность (faithfulness), фактическая корректность, отказ от ответа
- **LLM‑based оценка**: как использовать сами модели для автоматической оценки качества RAG
- **Инструменты вроде RAGAS**: готовые пайплайны для измерения context precision/recall, faithfulness, answer relevancy
- **Экспериментальный цикл**: как ставить A/B‑тесты и отслеживать влияние изменений в архитектуре RAG на метрики

> 💡 Цель оценки — не просто «получить число», а увидеть, на каком этапе пайплайна RAG теряется качество: поиск, генерация или стыковка между ними.

## 5. Агенты (Agentic AI)

### 5.2 Инструменты (tools)

Инструменты (tools) — это функции или внешние сервисы, которые AI‑агент может вызывать, чтобы получать данные и выполнять действия во внешнем мире: ходить в API, базу данных, CRM, почту, таск‑менеджер и т.д. Без инструментов агент остаётся «умным собеседником», а с инструментами он превращается в исполнителя задач, который умеет не только рассуждать, но и действительно что‑то делать.

### 5.3 Память

У базовой LLM вся «память» ограничена контекстным окном: модель видит только кусок переписки и забывает всё, что не попало в текущий запрос. Для AI‑агента этого мало — ему нужно помнить факты, решения и предпочтения между шагами и сессиями. Память агента — это набор механизмов (от краткосрочного состояния до векторных хранилищ и баз данных), которые позволяют системе быть последовательной и накапливать опыт.

- Почему нельзя просто сохранять весь лог диалога и важно думать про структуру и срок жизни памяти (TTL)

### 6.2 Мониторинг LLM: задержка, токены и стоимость

Когда LLM‑система выходит в продакшен, она перестаёт быть просто «моделью в ноутбуке» и превращается в бизнес‑сервис с бюджетом и SLA. Задержка (latency) определяет, насколько комфортно пользователю ждать ответа, использование токенов напрямую влияет на стоимость, а вместе они задают экономику всего продукта.

Если не следить за этими метриками, легко получить ситуацию, когда бот «вроде работает», но каждый диалог стоит слишком дорого или начинает отвечать с большой задержкой при росте нагрузки. Поэтому базовый мониторинг времени ответа, токенов и стоимости — обязательная часть эксплуатации любого LLM‑сервиса.

- Какие минимальные метрики нужно собирать для LLM‑сервиса: время ответа, токены, стоимость, частота ошибок
- Как по этим метрикам находить аномалии: рост стоимости диалога, деградацию скорости, всплески неуспешных запросов

### 6.3 CI/CD для LLM: регрессионные тесты промптов

В классической разработке CI/CD следит за тем, чтобы новые изменения в коде не ломали уже работающий функционал. ## 8. Безопасность и этика



### 8.1 Защита от prompt injection: валидация

Prompt injection — это атака, при которой злоумышленник вставляет в пользовательский запрос скрытые инструкции, чтобы переопределить поведение модели и заставить её выполнить произвольные действия вместо исходной задачи. В отличие от классических code injection атак, здесь эксплуатируется сама природа LLM: модель не различает системные инструкции разработчика и данные от пользователя, обрабатывая всё как единый текст на естественном языке.

OWASP Top 10 for LLM Applications 2025 ставит prompt injection на первое место среди угроз безопасности AI-систем. Защита строится не на уровне самой модели, а через архитектурные меры: валидацию входов, разделение системных и пользовательских инструкций, ограничение привилегий модели, песочницу для исполнения команд и мониторинг необычных паттернов в выводе.

#### Что нужно изучить на этом этапе

- Чем prompt injection отличается от jailbreaking и как эти атаки работают на практике.
- Почему LLM не может различать системные инструкции и внешние данные на уровне модели.
- Как правильно разделять system messages и user input, используя структурированные форматы (например, ChatML или JSON-схемы).
- Какие методы валидации и фильтрации входов снижают вероятность успешной инъекции: лимиты символов, whitelist-подход, детектирование опасных паттернов.
- Как ограничивать привилегии модели: принцип least privilege, изоляция критических операций, human-in-the-loop для рискованных действий.
- Как выстраивать мониторинг и обнаружение аномалий в выводе: фильтры на выход, детектирование попыток выполнения команд или утечки системных инструкций.

### 8.2 Обнаружение и удаление персональных данных (PII)

LLM-системы обрабатывают большие объемы текста, включая пользовательские запросы, логи, документы и результаты RAG-поиска. В этих данных регулярно встречаются персональные данные (PII): имена, адреса, телефоны, email, номера карт, паспортов и другие идентификаторы. Если такие данные попадают в логи, промежуточные хранилища или отправляются во внешние API без обработки, это создаёт риски утечки и нарушения требований защиты данных.

Обнаружение и удаление PII — это не просто фильтрация очевидных паттернов вроде номеров телефонов. Это многоуровневая задача: распознавание именованных сущностей (NER), контекстная классификация, маскирование или псевдонимизация перед отправкой данных в LLM, восстановление исходных значений при необходимости и соблюдение правил обработки данных (GDPR, 152-ФЗ). Правильно выстроенный pipeline защиты PII позволяет безопасно использовать внешние LLM и облачные сервисы, минимизируя риск утечки чувствительной информации.

#### Что нужно изучить на этом этапе

- Какие типы персональных данных встречаются в LLM-системах: имена, контакты, финансовые идентификаторы, медицинские данные, геолокация.
- Как работает автоматическое обнаружение PII: NER-модели, regex-паттерны, контекстная классификация токенов.
- Чем отличаются маскирование, псевдонимизация и анонимизация, и когда применять каждый метод.

### 8.3 Контроль расходов и защита от злоупотреблений

LLM-системы в продакшене работают по токенной модели тарификации: каждый запрос и ответ измеряется в токенах, а стоимость складывается из цены за входные (prompt) и выходные (output) токены. Без контроля расходов даже небольшая автоматизация может превратиться в источник неожиданных трат: длинные контексты RAG, reasoning-модели с многократным выводом, пользовательские запросы без ограничений и автоматические retry-логики способны быстро исчерпать бюджет или превысить лимиты провайдера.

Защита от злоупотреблений и контроль расходов — это не просто настройка квоты на уровне провайдера. Это многоуровневая система: лимиты на токены и запросы (tokens per minute, requests per minute), ограничения на уровне пользователя, команды и приложения, мониторинг бюджетов в реальном времени, обнаружение аномальных паттернов использования и fallback-стратегии для предотвращения отказа сервиса при исчерпании квот. Правильно выстроенный контроль помогает избежать как технических проблем (rate limit exceeded, timeout), так и финансовых — неконтролируемого роста стоимости LLM-инфраструктуры.

#### Что нужно изучить на этом этапе

- Как устроена токенная экономика LLM: разница между input и output токенами, почему reasoning-модели дороже, как учитывать стоимость контекста в RAG.
- Какие типы лимитов применять на практике: TPM (tokens per minute), RPM (requests per minute), concurrency caps, max_tokens per request, daily/monthly quotas.
- Как выстраивать многоуровневые квоты: per-key, per-user, per-team, per-app, per-organization и глобальные лимиты на уровне всей платформы.
- Как правильно возвращать HTTP 429 (Too Many Requests) с Retry-After и структурированными ошибками, чтобы клиенты могли корректно обрабатывать превышение лимитов.
- Как отслеживать использование в реальном времени: dashboard с usage metrics, предупреждения на 80% и 95% квоты, лог аномальных запросов и внезапных всплесков трафика.
- Как защищаться от abuse: детектирование ботов и автоматических скриптов, CAPTCHA для подозрительного поведения, блокировка ключей при превышении честных порогов использования.
